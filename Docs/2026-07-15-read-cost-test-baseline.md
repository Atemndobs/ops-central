## The open question this test answers

**72 MB burned on a day with 1 pageview is unexplained.** All 19 crons combined
account for only ~3 MB/day (audited — 18 of 19 are correctly bounded; the one
that wasn't is #269). The remaining ~69 MB has no attribution from source
analysis.

Leading hypothesis: **a browser tab or phone left open**. Reactive subscriptions
stay live and re-execute on every write to their read set — including the 288/day
notification cron's writes — generating reads with **no new pageview**, which is
invisible to PostHog.

Do **not** assume Jul 15's low total means the fixes worked. Usage collapsed the
same day, so the variables are confounded. Counter-evidence: **Jul 13 had only 3
pageviews and burned 2.4 GB — the highest day of the month.** Low usage does not
imply low reads. That is precisely what this test disambiguates.

## After the test — what to capture

1. **Dashboard → Usage → Database I/O**, note the Jul 15 total; subtract the
   ~72 MB pre-test figure to isolate the test's cost.
2. **Dashboard → Usage → Breakdown by function → `Database I/O` tab** (not the
   Function Calls tab). This is ground truth for per-function bytes and the only
   way to close the attribution gap. `npx convex insights` refuses deploy keys,
   so this needs a human on the dashboard.
3. `csoi analytics visits` — pageviews generated by the test, for a per-pageview
   cost.
4. `csoi perf --docs` — confirm `users` stays at ~523 B/doc and no table regressed.

Divide test-attributable bytes by the actions performed. A single assign→execute
→complete cycle costing tens of MB means a reactive-amplification problem still
live; costing a few hundred KB means the remaining floor is background, not
user-driven.
